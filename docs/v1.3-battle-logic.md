# V1.3 对战决策逻辑说明

> 描述 `20201227_v1.3.js` 相对 v1.2 的改动：换人回归修复(改动I) + softmax 候选质量(改动 M/K/J/L)。
> 决策控制流地图见 [docs/analysis/attemptCommand-control-flow.md](analysis/attemptCommand-control-flow.md)。
> 问题来源：v1.2 实测 7 case + 5 report(report 为人类亲自对战 v1.2 AI)。
> v1.0/v1.1/v1.2 既有逻辑见各自 battle-logic 文档。

---

## 背景

v1.2 上线后实测暴露：

- **P0 回归**：改动F(standard主导换人)在被迫/KO换人路径失效。`attemptSwitch` 的 goodAttackSwitch 只按输出筛选，standard 高但输出不够的墙(case7 超坏星 standard+35)进不了攻击候选，导致换上 standard 最低的送死(case4-R8、case7-R2)。主动换人路径正常(case6-R7)。
- **P1 softmax 候选质量**：① 溢出虚高(水炮158%名义伤害压过冲浪130%，无视溢出边际效应+命中率) ② 先制度完全没建模(case7-R3 子弹拳先制+1伤害最高却被随机抽到拍落，被秒) ③ 上下位招冗余(同属性下位招稀释概率) ④ 自损debuff招非首选时仍被抽中(case7 流星群)。

---

## 改动 I：被迫换人纳入高 standard 候选

**根因**：`attemptSwitch`(L1994 起)最终选择，当 goodAttackSwitch 非空、alternative/goodForSwitch 空时走 `pickBySwitchStandard(goodAttackSwitch, switchesList)`；而 pickBySwitchStandard 的 fallback 仅在 candList 为空时生效。高 standard 墙没进 goodAttackSwitch → 永不被纳入。

**修复**：仅在 `passive`(被迫/KO)换人、走到 goodAttackSwitch 分支时，把"全体存活里 standard 接近最高(容差50)且非被克制(>-50)"的单位**并入候选**，与攻击手一起由 standard 主导：

```javascript
if (passive) {
    // 求全体最高 standard，再收集 "本是攻击候选 或 standard>=最高-50 且 >-50" 的单位
    cswitch = pickBySwitchStandard(stdCand, switchesList);
} else {
    cswitch = pickBySwitchStandard(goodAttackSwitch, switchesList); // 主动换人不变
}
```

`!passive`(主动换人)路径完全不变，保 case6-R7。

**设计意图**：早期被动换人偏"抓机会输出"是因主动换人弱；现主动换人(E/F)已强，被动换人回归"兼顾攻防"。

## softmax 候选质量改动(M→K→J→L)

均在 attemptCommand 的 `if (!commandDecided)` softmax 段。执行链：第一遍算 smFinal(含 M 折减、K 加权) → J/L 标记 smDominated → 第二遍过滤(带守卫) → 加权采样。

辅助函数(块顶定义一次，J/L 共用)：`smAcc`(命中,>100视为100) / `smPrio`(先制) / `smDebuffRank`(自损等级: 特攻-2招→2, 蛮力/近战类→1, 其余→0)。

### 改动 M：溢出折减 + 命中率

**理念**：名义伤害有边际效应，超过对手当前 HP 的溢出几乎无价值。把 smFinal 伤害基底从名义 movepow 换成"有效伤害×命中率"：

```
ratio = smFinal / foeHp                              // 占对手当前HP
有效 = (ratio<1.10 ? ratio : 1.10) + max(0, ratio-1.10)*0.1   // 1.10内全额，溢出仅×0.1
       再折回绝对值: × foeHp
命中系数: accurcy 30~100 时按 L2834 算法(含对手闪避 statBoost(7))折算，否则 1.0
smFinal = 有效 × 命中系数
```

- **1.10 阈值**留 ±5% 伤害估算裕量(伤害估算偏差主要来自对手耐久，"看似刚好能秒"的招不被误判溢出)。
- 验证：冲浪(130%,命中100) eff≈112 > 水炮(158%,命中80) eff≈92 → 选冲浪。
- 仅作用 smFinal，不动 getMoveDamage 主逻辑(被多处调用)；不影响斩杀判断(走 commandDecided 用原始伤害)。

### 改动 K：先制度建模(速度评估为主驱动)

**理念**：先制价值本质来自速度劣势。速度分层为主，先制斩杀/对手能秒我为放大器。

```
速度分层(正先制招 priority>0):
  我方稳定更快(mySpd > 对手速度上界) → 不加权(反正先手，先制无额外价值)
  可能更慢                          → ×1.15
  稳定更慢(mySpd < 对手速度下界)     → ×1.30
放大器:
  先制可斩杀(movepow>=foeHp) 且 对手能秒我(lastFoeBestDmg>=我方当前HP%) → ×1.60
  先制可斩杀 或 对手能秒我 → 至少 ×1.30
```

`lastFoeBestDmg` 由 getFoeThreatToMe 暴露(对手对我最大伤害占满血比例)。验证：case7-R3 子弹拳(先制可斩杀+对手能秒我) ×1.6 → 83.7 碾压拍落 44。

**突袭(296)负优先**：我方稳定更快 且 存在另一招伤害更高且能斩杀对手时，smFinal ×0.3(保留小概率，非剔除)；否则正常参与(残血/拼血场景突袭先制斩杀仍有价值)。

### 改动 J：上下位招剔除(严格支配)

第一遍后，判定"A 被 B 上位支配"(全满足 → smDominated[A]=true)：

1. 同属性(sys.moveType 相同)
2. smFinal[B] >= smFinal[A] × 0.98(小容差；±5%偏差来自对手耐久，A/B 同向偏，相对比较稳)
3. smAcc(B) >= smAcc(A)
4. smPrio(B) >= smPrio(A)
5. smDebuffRank(B) <= smDebuffRank(A)
6. 完全等价时保留索引靠前者(movepow 降序=上位)，防互删

第二遍候选构建跳过被支配招。修同打击面冗余(冷冻干燥/冰冻光束；突袭/拍落在先制不同时不互相支配)。

### 改动 L：自损 debuff 招非首选剔除

自损招(SELF_SPATK_DROP_2 等)当 smFinal < 最高 且 存在非自损招 smFinal >= 它时 smDominated=true；首选(能斩杀=最高分)保留。复用 J 的 smDominated。修 case7 流星群非首选仍被抽中。

### 候选清空保险

J+L 理论上不剔除最高分招(永不被支配)，但加防御：第二遍后若候选为空，忽略 smDominated 重建一次。叠加已有 candidates===1 强制采用兜底。

## v1.3.1 修改项（patch，基于 feedback/v1.3 case1/case2）

> 对应文件 `20201227_v1.3.1.js`；checklist [feedback/v1.3.1/checklist.md](../feedback/v1.3.1/checklist.md)。

### 改动1：KO 换人走 passive 路径

**问题**：KO 换人（`poke(battle.me).isKoed()`）调用 `attemptSwitch(false)`，passive=false。但改动 I（被动换人纳入高 standard 候选）只在 `if(passive)` 内触发，故 KO 换人永不走改动 I，选了 goodAttackSwitch 里 standard 最低的送死（case1-R6 选 standard 5 达摩狒狒而非 35 勾帕路翁；case2-R6 选 standard -75 铁螯龙虾送死，直接导致败北）。

**修复**：`attemptSwitch(false)` → `attemptSwitch(true)`。KO 换人本就是被迫，应走 passive 路径触发改动 I。

### 改动2：被动换人纯 standard 候选

**问题**：改动 I 的 stdCand 构建里 `goodAttackSwitch.indexOf(sl2)!==-1` 无条件纳入全部攻击候选，把 standard=-130 的铁螯龙虾拉进候选池（case2-R0）。

**修复**：passive 时纯按 standard 收集候选（`>= bestAllStd-30 && > -50`），删除攻击候选无条件纳入。容差 50→30（与 pickBySwitchStandard 内部一致）。输出过低的墙由改动3③在 getSwitchStandard 内惩罚，此处信任 standard。

### 改动3：standard 虚高缓解

**问题**：丰蜜龙（草+龙）vs 爱管侍（超能+妖精）被妖精 2x 克制应 -50，但耐久加分 +30×2=+60 抵消还赚成 50。根因：getSwitchStandard 缺"候选对对手输出"维度（结构性，留 v1.4）+ 耐久加分过易触发 + L1752 indexOf 漏参死代码。

**小改三处**：
- ① L1752 `indexOf()` → `indexOf(j)`，激活危险属性 -20 分支
- ② 耐久加分 L1769-1772 收紧：+30→+18，阈值 0.3→0.22（双维 +36 < 克制 -50，不再抵消）
- ③ L1813 return 前加输出过低惩罚：`getMoveDamage(slot).maxmovepow / 对手满血` <0.18 罚 -30，<0.30 罚 -15。是改动2 纯 standard 的安全网。

### 改动4：回退改动 J（上下位支配剔除）

**问题**：改动 J 误删伏特替换（有换人特殊效果不该当下位招）。

**修复**：注释 J 支配判定 for 循环。**保留** smDominated 声明、第二遍守卫 `&& !smDominated[smi2]`、改动 L（自损非首选剔除）、候选清空保险——L 独立于 J（只读 smDebuffRank/smFinals/smMaxFinal，不读 J 写的状态），故回退 J 不影响 L。

### 推迟到 v1.4

- **问题5**（威胁评估未露本系招式）：露招时补"未露属性的本系"按露招数衰减（0.8/0.5/0.2）。并入 v1.4 getSwitchStandard 重构。
- **问题3 结构性**：getSwitchStandard 防御部分复用 getFoeDamageToMe（统一量纲）+ 加完整输出维度，standard 变防御+输出综合分。呼应 finalScore 加权模型愿景。
- **问题6**（threat 算法）：threat 与 bestDmg 量纲关系后续专门讨论优化。
