# V1.2 对战决策逻辑说明

> 本文档描述 `20201227_v1.2.js` 相对 v1.1.x 的换人决策重构，面向对战观察和调参使用。
> v1.2 聚焦换人：改动 F（standard 主导换人目标选择）、改动 E（被秒风险驱动主动换人）。
> v1.0/v1.1 的招式伤害、softmax、对手决策池等逻辑不变，见 [v1.1-battle-logic.md](v1.1-battle-logic.md)。
> 问题来源与根因分析见 [feedback/v1.1.1/analysis.md](../feedback/v1.1.1/analysis.md)。

---

## 背景

v1.1.1 实测 7 局暴露两类换人问题，checklist 总评"该换人的时候太少换了"：

- **换人选错目标**（case4/5/6）：`getGoodForSwitch()` 算出的 `standard` 评分（综合属性克制/免疫/耐久/HP/入场伤害）只用于过滤 slot，分数本身被丢弃；`attemptSwitch()` 在候选列表里 `sys.rand` 纯随机抽，不看 standard。导致 case6-R7 把 standard=+80 的西狮海壬和 -95（被格斗2×克制）的三首恶龙一起随机，抽中被克制的。
- **该换没换**（case7-R3/R7、case3）：三档换人路径的"鼓励换人"条件只覆盖①速度劣势+被克制 ②被弱化 ③输出不足，没有"对手能一击秒我"维度。case7-R3 火焰鸟满血、输出够、未被弱化，但对手捷拉奥拉高速能一击秒它，三条件都不触发→不换→被秒。

## 改动 F：standard 主导换人目标选择

### F-1 抽出 `getSwitchStandard(slot)`

把原 `getGoodForSwitch()` 循环体内的 standard 评分逻辑抽成独立函数 `getSwitchStandard(slot)`（行为完全等价），供 `getGoodForSwitch` 与 `attemptSwitch` 复用。返回该候选的换入评分，非法 slot 返回 -9999。

`getGoodForSwitch` 改为调用它，三档阈值（standard>100→ret1，>50→ret2）不变，对外返回的 slot 列表行为不变。

### F-2 新增 `pickBySwitchStandard(candList, fallbackList)`

按 standard 主导从候选列表选换入目标，仅在分数接近的候选间随机：

1. `candList` 空则用 `fallbackList`；都空返回 -1。
2. 对每个候选算 `getSwitchStandard`，取最高分 `bestStd`。
3. 收集"接近最高分（容差 30）且非被克制（standard > -50）"的候选窗口。
4. 窗口为空（全部 ≤ -50 被克制）→ 退回最高分单个（不随机，选最不坏）。
5. 否则在窗口内 `sys.rand` 随机（保留服务型 BOT 随机性）。

参数含义：容差 30 ≈ 一重属性抵抗（+20）量级；-50 ≈ 至少被一重克制（getSwitchStandard 里单个 2× 克制即 -50）的剔除线。

### F-3 / F-4 `attemptSwitch` 选择改造

`attemptSwitch` 最终选择（alternative / goodForSwitch / goodAttackSwitch）全部从 `sys.rand` 改为走 `pickBySwitchStandard`。攻击候选（goodAttackSwitch）传 `switchesList` 作 fallback——攻击候选全被克制时退回全体按 standard 选。

兜底分支（原 recordBug 前）也先 `pickBySwitchStandard(switchesList, [])` 按全体 standard 选最优，救 case4 类"最优候选分数低（+5<50）进不了任何列表"的场景。

### 修复效果
- case6-R7：bestStd=80，三首恶龙(-95) 不满足 `> -50` 被剔除 → 选西狮海壬。
- case4-R6：攻击候选全被克制 → fallback 到 switchesList 按 standard 选 → 银伴战兽(+5) > 音波龙(-65)。

## 改动 E：被秒风险驱动主动换人

### E-0 新增 `getFoeDamageToMe(moveType, power, category)`

估算对手某招（给定属性/威力/物理或特殊）对我方当前宝可梦的伤害比例（占我方总 HP）。复用 `getGoodForSwitch` 的威胁公式结构：

```
dmg = (2*对手lv+10)/250 * 对手maxStat(攻击) / 我方stat(防御) * power
    * 属性克制倍率 * (本系 ? 1.5 : 1)
返回 dmg / 我方totalLife
```

用 `maxStat`（对手最强估计）= 乐观假设对手满输出，威胁评估宁高勿低。

> 注：代码原有的 `foeInformation.getCurrentPossibleDamage()` 名称/注释易误解——它实际算"我方招打对手"，**不是**"对手打我方"。故本维度新建独立函数，不复用它。

### E-1 新增 `getFoeThreatToMe()`（返回 0-100）

评估对手当前宝可梦对我方当前宝可梦的"先手秒杀"威胁。因子求和后 clamp 到 0-100：

| 因子 | 分值 | 说明 |
|------|------|------|
| A 速度（先手概率） | 0~40 | 我方速度 < 对手最小速度 +40；落在区间内（可能后手）+22；我方必先手 0 |
| B 能否秒我 | 0~45 | bestDmg≥1.0 +45；≥0.8 +32；≥0.6 +15；≥0.45 +6 |
| C 我方残血放大 | 0~15 | HP<0.35 +15；<0.5 +8；<0.7 +3 |
| D 对手进攻强化 | 0~15 | max(物攻,特攻)等级 ≥2 +15；=1 +8 |

**B 的伤害来源**：
- 对手**已亮相攻击招**（move(i).num>0 且非变化招）→ 逐招调 `getFoeDamageToMe` 取最大。
- **无任何亮相攻击招** → 假设对手持一个本系招（type1/type2，威力 80，category 取对手物攻/特攻强项）兜底估算。亮相后即用实际招覆盖。

带 `print_s("getFoeThreatToMe threat:N bestDmg:X spd:AvsB")` 便于实战验证。

### E-2 三档换人路径接入

主循环外算一次 `foeThreat`，三档共用。在每档**安全阀（对手强化随机拒绝行）之后**追加，使被秒维度能覆盖"对手强化时保守不换"——case7 捷拉奥拉正是"对手强且能秒"，应果断换（与改动 A 风格一致）：

| 档 | 触发条件 |
|----|---------|
| 第一档 bestSwitchList | `foeThreat >= 55 && bestSwitchList.length > 0` → rejected=false |
| 第二档 goodSwitchList | `foeThreat >= 65 && (goodSwitchList.length>0 \|\| goodForSwitch.length>0) && sys.rand(0,3)` |
| 第三档 goodForSwitch | `foeThreat >= 75 && goodForSwitch.length>0 && !sys.rand(0,3)` |

**硬性前提**：被秒维度**永不在空候选时触发换人**——每档都带 `&& 候选列表非空`。场下没合适换入时威胁再高也不换，避免"换人白送"。阈值递增 + 必须有 good 候选 + 保留随机，三重约束防止面对 setup sweeper 反复送人头。

### 修复效果
- case7-R3：火焰鸟满血（C=0），但对手捷拉奥拉高速（A=+40）+ 等离子闪电拳对火焰鸟伤害≥1.0（实战 100% 一击秒，B=+45）→ threat≈85 ≥ 55，且 good=1（河马兽）→ 触发换人换走火焰鸟。

## 改动 G：修复主循环确定性决策被 softmax 覆盖

### 背景：两阶段决策架构

attemptCommand 的决策分两个阶段（详见 [docs/analysis/attemptCommand-control-flow.md](analysis/attemptCommand-control-flow.md)）：

1. **阶段1 主循环**（v1.0 时代逻辑）：逐招检查斩杀/先制/破替身/变化招/换人等特殊机制，命中即 `break` 出招。这里**有**先制、命中率、斩杀判断。
2. **阶段2 softmax**（v1.1 新增）：主循环没 break 时落到此，全量攻击招按 `finalScore=纯伤害` 加权采样。这里**只有伤害**，无先制/命中率/附加效果。

### 问题（v1.1.2 case3-2）

阶段1 的特殊 break 出来后 `rejected=false`，跳过换人，但**直接进入阶段2 softmax 无条件用纯伤害重算 usemove**，覆盖了主循环选好的招。

实测：象牙猪对 41% 血 +1 摔角鹰人，冰砾（先制，伤害 0.654）打印"先制斩杀生效"，但落入 softmax 后 `floor=251` 把冰砾(185)过滤，只剩冰柱坠击(386,非先制) → AI 点冰柱坠击 → 象牙猪先被吸取拳秒，冰砾没打出。日志矛盾：`先制斩杀生效` + `chosen=冰柱坠击`。

### 修改

新增 `commandDecided` 标志（主循环前声明）。所有"确定性决策 break"（斩杀/先制/破替身/破画皮/变化招 ret≥3/换人招/起死回生/豁命/接力棒/Z招 等，约 26 处）前置 `commandDecided = true`。softmax 段整体包在 `if (!commandDecided) { ... }` 内——确定性决策时跳过 softmax，沿用主循环的 usemove，仅走后续 Mega/Z 标记与 sendCommand。

**唯一不标记的 break** 是普通攻击招出口（`if (!rejected && choosetime>=len*3...) break`）和高伤害确认（damagePercent>1.5）——它们本就该交给 softmax 随机化。

> 注：这是过渡性止血。v1.3 统一 softmax 架构后，确定性决策将改用"高权重 finalScore"表达而非 break 跳过，commandDecided 标志届时移除。

## 改动 H：残局收敛到唯一最优解

### 问题（v1.1.2 case4-4）

对手只剩最后一只时无换人博弈，本应收敛到唯一最优解，但残局 T 衰减（v1.1.1）只降 T 没把候选压到 1。实测 `candidates=2 [交错闪电100,逆鳞90]` 仍在随机（虽然逆鳞也秒了，但不该有二选一）。

### 修改（thresh override）

softmax thresh 计算后追加：`if (foeInformation.getPokeCount() <= 1) smThresh = 0.97;`。对手只剩 1 只时 floor 拉到 maxFinal×0.97，只有 ≥97% 最高分的招通过，候选收敛到唯一最优解（真等价的同分招仍可多个，无妨）。`getPokeCount()` 返回对手存活总数（status≠31）。

## 调参点速查

| 参数 | 当前值 | 位置 |
|------|--------|------|
| pickBySwitchStandard 容差 | 30 | 越小越贪心，越大随机候选越多 |
| pickBySwitchStandard 被克制剔除线 | -50 | standard 低于此被排除出随机窗口 |
| 被秒维度阈值（三档） | 55 / 65 / 75 | 越低越易触发换人 |
| getFoeThreatToMe 假设本系招威力 | 80 | 对手未亮相时的兜底估算 |
| getFoeDamageToMe 对手攻击估计 | maxStat | 宁高勿低（乐观假设满输出） |
