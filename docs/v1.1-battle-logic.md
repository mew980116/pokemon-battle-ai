# V1.1 对战决策逻辑说明

> 本文档描述 `20201227_v1.1.js` 的完整决策行为，面向对战观察和调参使用。
> 在 v1.0 基础上新增：换人概率预测（`estimateFoeSwitchProb`）、温度参数真 softmax（改动 D v2）。
> 深度技术分析见 [docs/analysis/decision-logic.md](analysis/decision-logic.md)。

---

## 1. 每回合决策流程总览

```
onOfferChoice 触发
    │
    ├─ 我方宝可梦已 KO？→ attemptSwitch，结束
    ├─ 所有招式被禁用？→ attemptSwitch，结束
    ├─ 可用招式 < 2 且不可换人？→ 强制出最强招，结束
    │
    └─ 进入 attemptCommand 主决策
           │
           ├─ 1. 计算当前宝可梦各招式伤害分（getMoveDamage）
           ├─ 2. 评估场下换入候选（getGoodForSwitch × 3 档）
           ├─ 3. 主循环：逐招评估，决定出招或换人
           ├─ 4. [v1.1 新增] 温度 softmax（改动 D v2）：
           │       ├─ 估算对手换人概率（estimateFoeSwitchProb）
           │       ├─ 计算局面温度 T
           │       ├─ 按混合评分 + 战略动作构建候选集
           │       └─ 加权采样，最终选招
           └─ 5. 发送指令（sendCommand）
```

## 4. 换人概率预测（estimateFoeSwitchProb）【v1.1 新增】

扫描对手已知 bench 宝可梦，估算对手下一回合换人的概率（0-100），并返回最可能换入的 pokemon。

**评估因子**：

| 因子 | 方向 | 说明 |
|------|------|------|
| 我方最强招式对 bench 的效果 | 效果越低 → 换入意愿越高 | 免疫=6分，高抗=3分，中抗=1.5分，普通=0.8分，被我克制=0.4分 |
| bench 对我方的属性威胁 | 威胁越高 → 换入意愿越高 | 2× 或以上时分数倍增 |
| bench 当前 HP | HP 越低 → 换入意愿越低 | 用离场时 HP 估算；未出场过则视为满血 |
| 入场伤害（我方岩钉/地钉） | 伤害越高 → 换入意愿越低 | 岩钉按属性倍率折算；地钉对地面系免疫 |

**综合评分换算（典型值）**：

| 场景 | 估算 switchP |
|------|-------------|
| bench 完全免疫我方所有招 + 威胁我方 2× | 约 80-100 |
| bench 高抗我方主攻 + 轻度威胁 | 约 50-70 |
| bench 普通匹配，无明显优势 | 约 10-25 |
| 对手 bench 全部未亮相（pokeNum=0） | 0（无信息，不预测） |

**用途**：
- 驱动 softmax 中的分数混合比例
- UT/VS/Flip Turn 的得分加成权重
- 调整局面温度 T（高 switchP → T 上升，允许更多探索）

## 5. 温度 softmax（改动 D v2）【v1.1 大幅重构】

### 5.1 与 v1.0 的根本区别

| 对比项 | v1.0 | v1.1 |
|--------|------|------|
| 触发条件 | 2 招 movepow 之差 < 阈值 | 所有 movepow > 0 的招式**每回合都参与** |
| 阈值来源 | pool（对手信息量） | 局面温度 T（紧迫度） |
| 类型克制场景 | softmax 完全不介入 | 全量参与，弱招有合理概率 |
| 对手换人预测 | 无 | 混合"打当前"与"打换入"分数 |
| 换人招（UT/VS） | 仅靠 movepow 竞争 | foeSwitchP 加成 |
| 战略动作（岩钉） | 不参与 | 未布时 stratPow=70，与攻击招竞争 |

### 5.2 局面温度 T

T 控制采样的集中程度：T 低 → 贪心（几乎选最优），T 高 → 均匀探索。

**基础 T = 2.0**，根据以下条件调整：

| 条件 | 调整 | 效果 |
|------|------|------|
| 对手攻击强化 ≥ 3 | × 0.45 | 极度贪心：对手强化后行为已可预测，不需要随机 |
| 对手攻击强化 ≥ 2 | × 0.60 | 偏贪心 |
| 对手攻击强化 ≥ 1 | × 0.80 | 轻度收敛 |
| 我方 HP < 25% | × 0.55 | 残血下不能乱打 |
| 我方 HP < 45% | × 0.75 | 偏保守 |
| foeSwitchP > 65 | × 1.25 | 对手换人时多探索无妨（打到谁都无所谓） |
| foeSwitchP > 40 | × 1.10 | 轻度放开 |

**T 范围**：钳制在 [0.4, 4.0]。

**T 对概率的影响示例（两招 movepow 比 2:1）**：

| T | 强招概率 | 弱招概率 |
|---|---------|---------|
| 0.5 | 约 80% | 约 20% |
| 1.0 | 约 71% | 约 29% |
| 2.0 | 约 59% | 约 41% |
| 4.0 | 约 54% | 约 46% |

### 5.3 候选集构建与评分

**攻击招**（movepow > 0）：

```
finalScore = movepow_vs当前 * (1 - switchP/100)
           + movepow_vs换入 * (switchP/100)

movepow_vs换入 = movepow_vs当前 * (effIncoming / effCurrent)
  其中 effIncoming/effCurrent 为类型倍率比值（只调整属性差异，STAB/天气等不变）
```

特殊加成：
- **UT/VS/Flip Turn（369/521/838）**：`finalScore *= (1 + switchP * 0.008)`
  - switchP=60 时约 ×1.48，switchP=80 时约 ×1.64

**变化招（movepow = 0）**：
- **岩钉（446）**：未布时 stratPow = 70（直接参与竞争）；已布时不参与
- **地钉（191）**：岩钉已布时 stratPow = 40；岩钉未布时仅 18（岩钉优先）

**排除**：其他变化招（强化/睡眠/电磁波等）暂不参与 softmax，走原有确定性逻辑。

### 5.4 加权采样

```
weight[i] = finalScore[i] ^ (1/T)   // 幂函数，T 越大越均匀
normalized[i] = floor(weight[i] / max(weight) * 100)  // 最大权重归一为 100
从 normalized[] 按权重采样
```

**单候选时行为**：与 v1.0 完全相同，不改变结果。

## 7. 输出格式（调试日志）

v1.1 的 `softmax:` 行格式已更新：

```
softmax: pool=72 T=1.8 switchP=45(bench=248) candidates=3 chosen=1(急速折返) [急速折返,冲浪,暗黑波动]=[100,72,58]
```

**字段说明**：

| 字段 | 含义 |
|------|------|
| `pool=` | getOpponentDecisionPool 的值（仅供参考，不再控制采样宽度） |
| `T=` | 当前局面温度 |
| `switchP=` | estimateFoeSwitchProb 返回的换人概率 |
| `bench=` | 预期换入的 bench pokemon 编号（0 表示无已知 counter） |
| `candidates=` | 参与 softmax 的招式数量 |
| `chosen=` | 最终选中的招式槽及名称 |
| `[招式列表]=[权重列表]` | 各候选的归一化权重（最大=100） |

**softmax 在 v1.1 中每个"普通出招"回合都会输出**（包括单候选的 candidates=1 情况），便于追踪温度和 bench 评估值。

## 8. 已知待验证点（v1.1 新增）

以下行为需要实战确认，详见 [feedback/v1.1/checklist.md](../feedback/v1.1/checklist.md)：

1. **`hpPercentageWhenLeave` 的初始值**：bench pokemon 从未出场时，该字段是否为 0（代码当前将 0 视为满血处理）
2. **`sys.pokeType2` 单属性宝可梦**：是否返回 18（无属性），还是与 type1 相同；影响 `bt2 < 18` 判断

## v1.1.1 修改项

> 小版本补丁，基于 `feedback/v1.1/case 1.md` 和 `case 2.md` 实测问题。
> 对应文件：`20201227_v1.1.1.js`；checklist：[feedback/v1.1.1/checklist.md](../feedback/v1.1.1/checklist.md)。

### 修改 1：残局 T 衰减

**问题**：v1.1 的 T 计算没有"剩余宝可梦数"维度，残局（双方各剩 1-2 只）仍使用 T=2.0 的基础值，导致本该贪心的场面仍随机出弱招。

**修改**：在现有 T 调整之后、候选集构建之前，追加：

```javascript
var smTotalAlive = switchesList.length + 1;
if (smTotalAlive <= 2) smT *= 0.30;
else if (smTotalAlive <= 3) smT *= 0.45;
if (smT < 0.4) smT = 0.4;
```

`switchesList` 只含存活 bench（slot 1-5，非 KO），加 1 代表当前出战。

**日志字段新增**：`alive=N`，显示当前全队存活数，便于验证衰减是否触发。

### 修改 2：专爱系道具 T cap

**问题**：持有专爱头带/眼镜/围巾（item 4/5/6）被锁定招式后，不可换人，此时 T=2.0 仍允许随机出低分招式（case 1 武道熊师反复选毒击）。

**修改**：残局衰减之后追加：

```javascript
if (([4, 5, 6]).indexOf(poke(battle.me).item) !== -1 && switchesList.length === 0) {
    if (smT > 0.6) smT = 0.6;
}
```

仅在 `switchesList.length === 0`（无可换目标）时生效，有换人余地时不限制 T。

### 修改 3：finalScore 阈值过滤

**问题**：v1.1 全量 `movepow > 0` 的招式参与 softmax，低 T 时低分招式的绝对权重虽小但仍不为零，多回合累积后仍有概率被选中（case 1 T=1.1 时毒击仍有约 8% 选中率）。

**修改**：将候选集构建改为两轮遍历：

1. 第一遍：计算所有招式的 `finalScore`，记录 `smMaxFinal`
2. 计算过滤下限：`smFloor = smMaxFinal * smThresh`
   - T ≤ 0.8 → thresh = 0.85
   - T ≤ 1.5 → thresh = 0.75
   - T ≤ 2.5 → thresh = 0.65
   - T > 2.5 → thresh = 0.55
3. 第二遍：仅 `finalScore >= smFloor` 的招式进入候选集

**保证**：最优招式（finalScore = smMaxFinal）必然通过（1.0 ≥ thresh），候选集不会为空。

**日志字段新增**：`thresh=N` 和 `floor=N`，分别显示阈值比例和绝对下限值。

**v1.1 print_s softmax 行新格式（v1.1.1）**：

```text
softmax: pool=72 T=0.6 alive=2 switchP=12(bench=0) thresh=0.85 floor=85 candidates=1 chosen=0(暗冥强击) [暗冥强击]=[100]
```

## v1.1.2 修改项

> 小版本补丁，修复 v1.1.1 阈值过滤改动引入的回归 bug。
> 对应文件：`20201227_v1.1.2.js`；分析：[feedback/v1.1.1/analysis.md](../feedback/v1.1.1/analysis.md) 第 1 节；checklist：[feedback/v1.1.2/checklist.md](../feedback/v1.1.2/checklist.md)。

### 修改：candidates=1 时强制采用唯一候选

**问题**：v1.1.1 的 finalScore 阈值过滤可能把候选集收窄到只剩 1 个招式（残局低 T、thresh=0.85 时高频）。但加权采样分支的条件是 `smCandidates.length > 1`，当候选只剩 1 个时该分支被跳过，`usemove` 不被更新，残留主决策循环最后一次迭代的 slot（可能是次优招）。

**实测表现**（case7-R14）：多龙巴鲁托对捷拉奥拉，流星群可一击秒杀（damagePercent 1.04），floor 过滤后候选集只剩流星群，但日志却是 `candidates=1 chosen=1(暗影球) [流星群]=100` —— 打出了次优的暗影球。一度被误认为专爱锁招，但该精灵此回合刚因 KO 重新上场（四招齐全），并未锁招。

**修改**：在采样分支前补 `=== 1` 强制分支（`20201227_v1.1.2.js` softmax 段）：

```javascript
if (smCandidates.length === 1) {
    usemove = smCandidates[0];            // 唯一候选直接采用
} else if (smCandidates.length > 1 && smTotal > 0) {
    // ...原加权采样逻辑不变
}
```

**影响面**：残局低 T 时候选集极易收窄到 1 个，该 bug 可能在残局每个回合触发，使 AI 持续打出主循环残留的次优招而非评分最优招。修复后 `candidates=1` 时 `chosen` 必然指向候选集中那个唯一招式，与 `[...]` 方括号显示一致。
