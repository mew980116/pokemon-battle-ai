# V1.0 对战决策逻辑说明

> 本文档描述 `20201227_v1.0.js` 的完整决策行为，面向对战观察和调参使用。
> 深度技术分析（含代码行号）见 [docs/analysis/decision-logic.md](analysis/decision-logic.md)。

---

## 1. 每回合决策流程总览

```
onOfferChoice 触发
    │
    ├─ 我方宝可梦已 KO？→ attemptSwitch，结束
    ├─ 所有招式被禁用？→ attemptSwitch，结束
    ├─ 可用招式 < 2 且不可换人？→ 强制出最强招，结束
    │
    └─ 进入 attemptCommand 主决策
           │
           ├─ 1. 计算当前宝可梦各招式伤害分（getMoveDamage）
           ├─ 2. 评估场下换入候选（getGoodForSwitch × 3 档）
           ├─ 3. 主循环：逐招评估，决定出招或换人
           ├─ 4. softmax 采样（改动 D）：对接近最优的候选招式随机选取
           └─ 5. 发送指令（sendCommand）
```

---

## 2. 招式伤害评分（getMoveDamage）

每回合开始时对当前场上宝可梦的 4 个招式评分，得到 `movepow[]` 数组。

**评分公式（简化）**：

```
movepow = 基础威力 × 属性相性 × STAB × 天气 × 场地 × 壁垒 × 特性 × 精准度折扣
```

主要修正项：
- 属性相性：0.25×～4×
- STAB：1.5×（特性「适应力」为 2×）
- 天气：晴天火系 1.5×/水系 0.5×，雨天反之
- 场地：草/电/心灵/迷雾各 1.3×～2×
- 精准度 < 100% 时按命中率打折

**自损招式惩罚（改动 B）**：

若无法 OHKO 对手（乐观估算），对以下招式按 `getOpponentDecisionPool()` 动态折扣：

| 招式类型 | 代表招式 | pool ≥ 70 | pool 40-69 | pool < 40 |
|---------|---------|-----------|-----------|-----------|
| 特攻 -2 | 流星群、过热、精神突破、叶片风暴、花朵加农炮 | × 0.60 | × 0.70 | × 0.85 |
| 攻击 -1 防御 -1 | 蛮力 | × 0.70 | × 0.80 | × 0.90 |
| 防御 -1 特防 -1 | 近身战、龙升天 | × 0.75 | × 0.85 | × 0.92 |

能 OHKO 时不惩罚，直接使用。

## 3. 换人候选评估（getGoodForSwitch）

对场下每只宝可梦打 `standard` 分（基础 25 分起），按属性相性、入场伤害、对手强化等加减分，得出三档候选：

| 档位 | 门槛 | 含义 |
|------|------|------|
| bestSwitchList | standard > 100 | 强势换入：属性抗性 + 对手已虚弱/中毒等多重加分 |
| goodSwitchList | standard > 50  | 中等换入：有属性优势但条件一般 |
| goodForSwitch  | 剩余            | 无明显优势但可接受 |

**对手强化对换人评分的影响**：对手攻击类强化 > 0 时 `standard` 扣 20-35 分，强化越高换入候选质量下降越大。

## 4. 主决策循环

循环最多运行约 `enabledAttackSlot.length × 4` 次，每次从招式列表中选一个候选（`usemove`），通过多层条件判断是否接受（`rejected = false`）。

**两阶段遍历**：
- `choosetime < length × 3`：**顺序遍历**——按伤害从高到低依次考察，保证每招至少被评估一遍
- `choosetime ≥ length × 3`：**随机遍历**——从招式列表随机抽取，引入不确定性

**接受条件（满足任一即接受）**：

| 情形 | 条件 |
|------|------|
| 先手斩杀 | 我比对手快 + 伤害 > 当前 HP |
| 优先度斩杀 | 先制技（优先级 > 0）+ 伤害 > 当前 HP |
| 后手斩杀 | 对手 HP < 50% + 伤害 > 130% HP |
| 伏特替换/急速折返 | 需要换人时 |
| 普通出招 | 伤害足够（> 50% 并 50% 概率，或 > 70%）|
| 变化招 | 有效度评分 ≥ 3 |

**换人触发**：接受换人后，进入三档换人路径决定换哪一只。

---

## 5. 换人三档路径

三档为互斥 else-if，进到哪档取决于场下候选质量：

### Path 1（bestSwitchList，最激进）

典型触发：场下有强势候选（standard > 100），且满足下列任一条件：
- 我方被弱化 + 打不死（< 80% HP）
- 纯粹打不死（< 60% HP）

**改动 A 新增**：对手攻击类强化总和 ≥ 2 且我方打不死（< 90% HP）时，主动鼓励换入强势候选。

硬拒条件（即使上述条件满足也不换）：
- 我方有替身：2/3 概率保留（替身是昂贵资产）
- 对手防御/特防/回避强化 > 3：对手在自保，不是在威胁我方，换人无意义

### Path 2（goodSwitchList，中等保守）

阈值更高（< 40% HP 才考虑换），且换人条件引入额外随机性。对手多项强化（> 2）时 2/3 概率拒绝——中等候选可能被已强化的对手秒掉。

### Path 3（goodForSwitch，最保守）

对手只要有任何攻击强化（> 1）就硬拒——场下没有强势候选，送人去喂对手毫无意义。

## 6. 对手决策池（getOpponentDecisionPool）

返回 0-100 的整数，量化对手当前可选空间大小，作为改动 B（自损惩罚）和改动 D（softmax 候选集宽度）的共享信号。

**各因子与典型 pool 值**：

| 因子 | 方向 | 最大影响 |
|------|------|---------|
| 对手存活数（≥5 只） | + | +30 |
| 未亮相宝可梦数（5 只未知） | + | +11 |
| 当前宝可梦未知招式数（4 招未知） | + | +6 |
| 道具未知 | + | +1 |
| 特性未窄化 | + | +1 |
| 对手攻击强化（+2 以上） | − | −25 |
| 对手 HP < 25% | − | −15 |
| 对手仅剩 1 只 | − | −20 |

**典型值参考**：

| 局面 | 估算 pool |
|------|----------|
| 开局（第 1 回合，对手信息全未知） | 75-80 |
| 中盘（存 4 只，部分招式已知） | 55-65 |
| 终盘（存 2 只，信息大量已知） | 30-45 |
| 对手已 +2 强化 | 上述各值 −25 |

## 7. softmax 招式采样（改动 D）

主循环结束、发送指令前，对伤害接近最优的候选招式做加权随机采样，替代固定取最高分的行为。

**候选集门槛**（由 pool 决定）：

| pool | smThreshold | 含义 |
|------|-------------|------|
| ≥ 70 | 75% | 宽松：最优分的 75% 以上均入选 |
| 40-69 | 85% | 标准：最优分的 85% 以上均入选 |
| < 40  | 90% | 收窄：最优分的 90% 以上均入选 |

**采样方式**：以各候选 `movepow` 为权重做线性加权随机（`sys.rand`）。

- 候选只有 1 个时：与原逻辑完全等价，不改变结果。
- 候选 ≥ 2 个时：伤害更高的招式被选中概率更高，但次优招式也有合理概率被选中。

**常见触发场景**：拥有两个同类型 STAB 招式（如冲浪+水炮），或两个不同类型但威力相近的招式，对中性目标时两者 movepow 接近，会触发 softmax。

## 8. 输出模式（调试日志）

用 `mew's` 账号对战时，`print_s` 日志全部输出到 PO 脚本控制台。

**关键日志行**：

```
enabledAttackSlot:0,1,2,3           ← 按伤害降序排列后的可用招式槽
movepow:135.2,120.4,70.1,0.0        ← 对应招式的伤害分
maxmove:0 135.2                      ← 最优招式槽及其分值
bestSwitchList: goodSwitchList:2,3 goodForSwitch:2,3
foe:0.60                             ← 对手 HP 比例
me:0.92                              ← 我方 HP 比例
冲浪:power 135.2 accurcy 100 damagePercent 0.52   ← 每招评估详情
softmax: pool=78 threshold=75 candidates=2 chosen=1(暗黑波动) scores=135,120
sendCommand();choice.type=attack,slot=1,...       ← 最终指令
```
