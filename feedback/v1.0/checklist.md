## A. 机制验证（每项需至少触发 1 次并确认行为正确）

### A1. getOpponentDecisionPool

- [ ] 开局第 1 回合 pool 在 70-80 之间（对手全未知）
- [ ] 对手被 KO 2 只后 pool 明显下降（约 10-15 分）
- [ ] 对手使用剑舞/蝶舞后，下一回合 pool 下降约 25 分
- [ ] 对手 HP < 25% 时 pool 下降约 15 分
- [ ] pool 值始终在 [0, 100] 范围内，未见负数或超百

**验证方式**：在日志中找 `softmax:` 行，观察 `pool=` 字段随局势变化。

### A2. 自损招式惩罚（改动 B）

- [ ] 持有流星群的宝可梦**进场首回合**（对手信息未知，pool 高）：流星群出现概率低于其他 STAB 招式
- [ ] 流星群能 OHKO 时：AI 正常使用，无惩罚（需确认 KO 发生）
- [ ] 对手残血（pool 低，coeff 为 0.85）时：流星群与普通招式竞争更均衡
- [ ] 近身战（SELF_DEF_DROP 类）：惩罚系数比流星群更轻，偶尔被选中

**验证方式**：观察含流星群/过热的宝可梦的招式分布，与无惩罚版本对比。

### A3. Path 1 强化换人（改动 A）

- [ ] 对手使用两次剑舞（攻击强化 = +2），AI 场下有强势候选（standard > 100）时，换人率明显提升
- [ ] 对手强化但我方能 OHKO（`maxDamagePercent >= 0.9`）时：AI 不换，直接打
- [ ] Path 2 / Path 3 场景（场下无 standard > 100 候选）：此逻辑不触发，换人率不受影响

**验证方式**：观察对手剑舞后的回合，对比有无强势候选时的换人决策。

### A4. softmax 采样（改动 D）

- [ ] 同一局面重复触发（相同血量、相同招式），出现不同的招式选择（随机性确认）
- [ ] `scores=` 字段与 `movepow:` 日志行的数值对应（确认 smScores 来源正确）
- [ ] `chosen=(招式名)` 输出的招式名可读、正确（确认 `sys.move` 查表正常）
- [ ] 单招式候选时（candidates=1）：不触发 softmax，行为与原逻辑相同

### A5. 未知信息因子（pool 的招式/特性/道具部分）

- [ ] 对手首次上场（0 招式使用）vs 已使用 2 招式后：pool 差值约 4-5（对应 `unknownMoveBonus` 4→2 的差）
- [ ] 对手道具被确认后（如确认携带专爱围巾）：pool 减 1
- [ ] `fpoke(battle.opp).pokemon.move(i).num` 行为确认：
  - [ ] 对手未使用过的招式槽 num 返回 0（未知）→ 因子正常工作
  - [ ] 对手未使用过的招式槽 num 返回非零（PO 全量暴露）→ 因子永远 0，需改为追踪已用招式数量

## B. 效果评估（需 ≥ 20 局，主观 + 客观结合）

### B1. 可预测性

- [ ] 玩家能否在 3 回合内准确预判 AI 下一招？（改动 D 目标：让玩家难以稳定预判）
- [ ] AI 是否出现"明显不合理"的随机选择（如在能 OHKO 时选了弱招）？

### B2. 自损招式使用频率

- [ ] 流星群类招式在全局的使用比例是否合理（约 30-50% 的"应该用的场合"被使用）？
  - 过低（< 20%）→ 惩罚系数过重，建议上调（0.70 → 0.75 / 0.80）
  - 过高（> 70%）→ 惩罚不足，建议下调（0.70 → 0.65）
- [ ] 流星群使用后特攻 -2，下一回合是否出现"明显弱招"被强行使用的情况？

### B3. 强化后换人反应

- [ ] 对手完成 +2 强化后，AI 换入率是否相比之前提高？
- [ ] 被换入的候选是否确实对当前对手有威胁（而非随机换）？

### B4. 整体对战体验

- [ ] AI 是否在应该保守（对手高强化、己方残血）时表现出更收敛的决策？
- [ ] AI 是否在对局早期（信息未知、双方均健康）表现出更多样的出招选择？

## E. Bug 记录模板

发现 bug 时按以下格式记录（新建 `bug_NNN.md` 或直接追加到此文件）：

```
**Bug #001**
- 触发条件：
- 观察到的行为：
- 预期行为：
- 相关日志（截取关键行）：
- 推测代码位置：
```

**Bug #001：softmax 触发率过低**
- 触发条件：对手具有属性克制优势时（AI 有 2× 或以上的克制招式），movepow 比值直接跌至约 25-75% 区间，低于任何阈值门槛，softmax 完全不介入。实测 case 1/2 共约 40 个决策回合，仅触发 3 次（均为近等势场景）。
- 观察到的行为：softmax 仅出现 3 次，大部分决策没有使用 softmax 采样，AI 在有克制招时几乎 100% 选择该招，行为高度可预测。
- 预期行为：softmax 应在"AI 具有明显优势招式但不应总选它"的场合也能介入，使对手无法稳定预判出招选择。不要求均等随机，但克制招的选中概率应从 100% 降低到 75-85%，留出约 15-25% 概率选择次优招式。
- 根因分析：当前阈值模型（pool≥70 用 75%，40-69 用 85%，<40 用 90%）只覆盖"近等势招式"场景。属性克制产生 2× 伤害差，movepow 比值约为 50%，远低于最宽松的 75% 门槛；4× 克制时比值约 25%。最需要随机化的场景（AI 有明确克制招）恰好是 softmax 完全不介入的场景。
- 数据依据（case 1）：
  - T01/T02：ratio=82%，pool=78/77，threshold=75，**触发**（近等势：冲浪 vs 暗黑波动）
  - T03：ratio=82%，pool 降至 threshold=85，82%<85%，**未触发**
  - T07-T17：ratio 均在 50-75% 区间（128.4/246.3=52.1% 等），**全部未触发**
  - T12/T14/T16：ratio≈75.0%（246.3×100 vs 328.4×75，浮点差零点几），**差边触发但未触发**
  - T19：ratio=94.8%，pool=48，threshold=85，**触发**
- 相关日志（截取关键行）：
  ```
  movepow:328.4,246.3,0.0,0.0    <- T07，ratio=75.0%，threshold=75，差值导致未触发
  movepow:246.3,128.4,0.0,0.0    <- T11，ratio=52.1%，远低于 75%，未触发
  softmax: pool=78 threshold=75 candidates=2 chosen=1(暗黑波动) scores=135,110   <- T01，触发示例
  ```
- 推测代码位置：`20201227_v1.0.js` softmax 块 `smThreshold` 判断与候选集筛选（`movepow[smi] * 100 >= maxmovepow * smThreshold`）。
- 候选修复方案：
  - **路线 A（快）**：将阈值降至 50/65/75，使克制差距场景也能进入候选集；实现简单但阈值选取需实战调参
  - **路线 B（彻底）**：改用带温度参数的真 softmax（对所有非零 movepow 加权采样），覆盖"有克制招但偶尔不出"的核心场景；实现成本更高，但随机化效果更自然
