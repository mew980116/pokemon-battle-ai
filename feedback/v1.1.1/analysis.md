## 1. softmax `candidates=1` 时选错招【v1.1.2 紧急修】

### 批注原文（case 7 第 761 行）

> `## 为什么选择的是暗影球而不是流星群，套了专爱了`

### 对战上下文（case7-R13→R14）

多龙巴鲁托因 KO 重新上场对捷拉奥拉，决策日志（case7 第 755-760 行）：

```text
me=100% foe=60% sw:best=0 good=0 for=0 pow=[流星群:317.41,暗影球:196.48,喷射火焰:147.11,十万伏特:0]
流星群:power 317.41 accurcy 90 damagePercent 1.04   ← 可一击秒杀满血对手
暗影球:power 196.48 accurcy 100 damagePercent 0.64
softmax: pool=63 T=0.6 alive=2 switchP=9 thresh=0.85 floor=269 candidates=1 chosen=1(暗影球) [流星群]=100
```

矛盾点：评分阶段流星群 = 100（最优），floor=269 过滤后**候选集只剩流星群（`candidates=1`，`[流星群]=100`）**，但最终 `chosen=1(暗影球)`，打出了次优的暗影球。

### 排除"专爱锁招"假设

多龙巴鲁托此时**没有被锁招**：

- R9 末上场 → R10 出暗影球 → **R11 被收回**（case7 第 633 行 `Mew's收回了多龙巴鲁托！`）。收回时专爱锁招解除。
- R13 因 KO **重新上场**（第 745 行），候选 `pow=[流星群, 暗影球, 喷射火焰, 十万伏特]` **四招齐全** → 证明此刻未锁招（若锁招，非锁定招会被引擎从 `enabledAttackSlot` 移除）。

所以这不是专爱，是真 bug。

### 代码根因（已核实）

`20201227_v1.1.1.js:3817` 的加权采样分支：

```javascript
if (smCandidates.length > 1 && smTotal > 0) {   // ← 只有候选 > 1 才执行
    var smPick = sys.rand(0, smTotal);
    var smCum = 0;
    for (var smr = 0; smr < smCandidates.length; smr++) {
        smCum += smScores[smr];
        if (smPick < smCum) { usemove = smCandidates[smr]; break; }
    }
}
```

当 v1.1.1 的 finalScore 阈值过滤把候选集收窄到**只剩 1 个**（`smCandidates.length === 1`）时，这个分支被跳过，**`usemove` 不被更新**。

`usemove` 进入 softmax 段之前的值，是主决策循环（`20201227_v1.1.1.js:2952` 起）最后一次迭代残留的 slot —— 本局即暗影球（slot 1）。于是日志出现 `candidates=1`（候选只有流星群 slot 0）却 `chosen=1(暗影球)` 的矛盾。

**这是 v1.1.1「阈值过滤」改动自身引入的回归**：过滤前候选集恒为多个（全量攻击招），`>1` 分支总会执行；过滤后可能只剩 1 个，旧分支条件不再覆盖。

### 影响面

残局低 T（T≤0.8 → thresh=0.85）时，floor 很高，候选集极易收窄到 1 个 —— 也就是说**残局几乎每个回合都可能触发**，AI 残局可能持续打出主循环残留的次优招而非评分最优招。比换人类问题更高频。

### 修复方案

在采样分支前补 `=== 1` 的强制分支：

```javascript
if (smCandidates.length === 1) {
    usemove = smCandidates[0];            // 唯一候选直接采用，修复 usemove 残留
} else if (smCandidates.length > 1 && smTotal > 0) {
    var smPick = sys.rand(0, smTotal);
    ...
}
```

最小改动、与现有结构对称、不影响 `>1` 的采样路径。

## 2. 换人选错目标——丢弃 standard 评分【v1.2】

### 批注原文

- case4-R6（第 392 行）：`## 场下有明显更好的妖精银伴战兽，不知道是多属性攻击没有识别为妖精属性还是什么原因没有换上`
- case5-R1（第 114 行）：`场下有漂浮特性免疫地震的洛托姆，但是换了烈焰马白给了`
- case6-R7（第 369 行）：`## 这里换西狮海壬应该是显著好于三首恶龙，因为对面不克制西狮海壬而西狮海壬有高伤害招式，三首恶龙换上来收益太低了，还被克制`

### 关键观察：评分是对的，选择没用评分

`getGoodForSwitch` 输出的 standard 评分（含防御、被克制、免疫维度）**算得正确**，但最终换上的不是评分最高的：

| Case | standard 最高（应换） | 实际换上 | 实际换上的 standard |
|------|---------------------|---------|--------------------|
| case4-R6 | 银伴战兽 **+5** | 音波龙 | **-65** |
| case6-R7 | 西狮海壬 **+80 / +35** | 三首恶龙 | **-95** |

（case6-R7 订正：对手是闪电鸟-伽勒尔=格斗/飞行系，威胁招雷鸣蹴击/勇鸟为格斗系。三首恶龙=恶/龙被格斗 2× 克制；西狮海壬=水/妖精抵抗格斗。评分 +80 vs -95 正确反映了这一点。）

### 代码根因（已核实）

`attemptSwitch`（`20201227_v1.1.1.js:1800`）的最终选择逻辑：

1. `goodForSwitch`（"好的防守选择"，来自 standard 评分）经常为空。
2. 真正决定换谁的 `goodAttackSwitch`（"好的攻击选择"，`20201227_v1.1.1.js:1827-1861`）的筛选条件**只看 `maxmovepow`（输出）+ 速度 + `maxDamagePercent`**，完全不参考 standard（防御/被克制/免疫）。
3. 最终选择（`20201227_v1.1.1.js:1867-1871`）：若 `alternative`（= goodAttackSwitch ∩ goodForSwitch）非空则在其中随机；否则在 `goodForSwitch` 或 `goodAttackSwitch` 里 **`sys.rand` 随机抽**。

后果：
- case6-R7：西狮海壬(+80) 和三首恶龙(-95) 都因"有高伤害招"进 `goodAttackSwitch=[1,4]`，然后随机抽中被克制的三首恶龙。
- case4-R6：银伴战兽(+5) 输出不够，**根本没进** `goodAttackSwitch=[3]`，于是只能换 standard -65 的音波龙。
- case5-R1：同理换上被地震秒杀的烈焰马，没换免疫地震的洛托姆。

**一句话**：`attemptSwitch` 把算好的 standard（防御/克制/免疫）评分丢掉了，只在"输出够高"的候选里随机。

### 修复方向（v1.2 细化）

- 最终 `cswitch` 不应在高输出候选里纯随机，而应让 standard 主导：在候选集里按 standard 排序/加权，被克制（standard 显著为负）的候选大幅降权或剔除。
- 同时保留必要随机性（服务型 BOT 定位），但限定在 standard 接近的候选之间。
- 附带核实：多属性宝可梦（银伴战兽随碟片改变属性）的属性是否被正确识别为当前属性（case4 用户疑点）。

## 3. 该换人却没换【v1.2，需新建 getFoeThreatToMe()】

### 批注原文

- case7-R3（第 240 行）：`## 对手捷拉奥拉的威胁巨大，场下有非常适合换的河马兽，但是没有换`
- case7-R7（第 452 行）：`## 妙蛙花对河马兽威胁巨大，场下有非常适合换的钢铠鸦，也没有换`
- case3-R1（第 64-65 行）：`## 对手的威胁巨大且高速，电磁波的收益更高` / `## 仅强化攻击/特攻的招式在对手至少一个本系克制我方时很危险`
- checklist 总评（第 104 行）：`该换人的时候太少换了`

### 对战上下文

- case7-R3：火焰鸟(满血)对捷拉奥拉(高速电系大威胁)，`good=1` 已检测到换人候选（河马兽免电），但 `switchP=1`，AI 选诡计 → R4 满血被等离子闪电拳一击秒杀。
- case7-R7：河马兽对妙蛙花，`good=1`（钢铠鸦），AI 选隐形岩 → 被诅咒强化压制。
- case3-R1：劣势高速对位（暗黑酋雷姆本系克制），AI 选诡计自强化 → R2 被秒。

### 代码根因（已核实）

换人三档路径（`20201227_v1.1.1.js:3230-3315`）的全部 `rejected=false`（鼓励换人）条件，只覆盖三个维度：

1. 我方速度劣势 **且** 被对手克制（type chart）；
2. 我方被弱化（statBoost 总和为负）；
3. 我方输出不足（`maxmovepow / foeMaxHp` 偏低）。

**完全没有"对手能否一击秒掉我方"这个维度**。case7-R3 的火焰鸟满血、输出也够、未被弱化，三个条件都不触发，所以 AI 不换 —— 即便对手一招就能秒掉它。

这正是 TODO 中长期挂起的 `getFoeThreatToMe()` 缺失（见 `TODO.md` 5.5 节、`docs/analysis/battle-relationships.md`）。

### 修复方向（v1.2 细化）

- 新建 `getFoeThreatToMe()`（返回 0-100），因子：
  - 对手是否确定/可能比我快（`foeInformation.getPossibleSpeed`）；
  - 对手**已亮相招式**中能 OHKO / 2HKO 我方的（只用已知信息，不做属性推断——遵循 battle-relationships 原则）；
  - 我方当前 HP 比例；对手进攻强化等级。
- 当 threat 高（对手大概率先手秒我）**且** 场下有 good 换入候选时，在三档路径追加 `rejected=false` 鼓励换人。
- 与 case3-R1 的"劣势对位不应纯自强化 setup"联动：高 threat 时也应抑制纯强化招的 softmax 权重。

## 4. 招式机制缺失【v1.2+ / 后置】

### 4.1 祸不单行未识别对手异常状态翻倍（case5-R9，第 492 行）

> `祸不单行的伤害计算错误，少算了因对手烧伤（进入异常状态）威力翻倍`

祸不单行（Lash Out）在对手处于 major status 时威力翻倍。当前 `getMoveDamage` 的威力公式未识别此机制，导致对手已烧伤后该招价值被低估。属 Game Rules / Reasoning 层补丁。

### 4.2 残局拖延价值 + "我方数量 > 对方且对方剩 1 只"识别（case2-R14，第 773-774 行）

> `## 这里没有什么适合换的，而且对面也就最后一只了，应该不要换人上去挨打才对，使用祈愿守住拖对手剧毒消耗最优，大不了把水伊布送掉换上来收残。`
> `## 问题：没有判断祈愿在对手中剧毒下的高收益，没有判断我方精灵多对方只有一只时可以送场上的精灵`

对手只剩 1 只且已中剧毒时，AI 换新精灵上去挨打（酋雷姆被月亮之力 4× 秒），而最优解是"祈愿+守住"拖剧毒消耗；要弃子也应送场上精灵而非换新精灵。缺两个识别：① 拖延招（守住/祈愿）在对手中毒局面的高收益；② "我方存活数 > 对方 且 对方剩 1 只"的局面信号（此时送场上精灵代价最低）。

工程量较大（涉及拖延价值评估、弃子逻辑），建议后置。
