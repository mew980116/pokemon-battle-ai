## A. v1.1.1 新增机制验证

### A1. 残局 T 衰减

**触发条件**：我方全队存活宝可梦 = 2 只（switchesList.length+1 = 2）或 = 3 只。

- [ ] 剩余 2 只时，日志 `alive=2`，T 约为基础值 × 0.30（如基础 T=2.0 → 约 0.60）
- [ ] 剩余 3 只时，日志 `alive=3`，T 约为基础值 × 0.45（如基础 T=2.0 → 约 0.90）
- [ ] 全队 6 只时，日志 `alive=6`，T 不受此项影响
- [ ] `alive=` 字段始终与实际场上局面一致（当前出战 + 存活 bench 数之和）

**验证方式**：找对战后期回合（己方只剩 2-3 只），对比 `T=` 字段与理论值。

### A2. 专爱系道具 T cap

**触发条件**：我方持有专爱头带/眼镜/围巾（item 4/5/6）且 `switchesList` 为空（无可换宝可梦）。

- [ ] 满足条件时，日志 `T=` 不超过 0.6
- [ ] 持有专爱但 **有** 可换宝可梦时：T cap 不生效，T 正常计算
- [ ] 不持有专爱时：T cap 不生效
- [ ] cap 与残局衰减叠加时（全队最后1只 + 专爱）：T ≤ 0.6，且通常等于 0.6

**典型场景（case 1 复盘）**：
- 武道熊师 Choice Band，只剩自己一只 → T 应 ≈ 0.60
- 此时 thresh=0.85，暗冥强击（100）vs 毒击（28） → 毒击被过滤，只选暗冥强击

### A3. finalScore 阈值过滤

**验证阈值与候选集的关系**：

| T 范围 | thresh | 典型效果 |
|--------|--------|---------|
| T ≤ 0.8 | 0.85 | 只允许 finalScore ≥ max×85% 的招式入选 |
| T ≤ 1.5 | 0.75 | 只允许 ≥ 75% |
| T ≤ 2.5 | 0.65 | 只允许 ≥ 65% |
| T > 2.5 | 0.55 | 只允许 ≥ 55% |

- [ ] 日志 `thresh=` 字段与上表对应（T 值 → thresh 值）
- [ ] 日志 `floor=` 字段 ≈ maxFinalScore × thresh（手动验算）
- [ ] 紧张局面（T≤0.8）时 `candidates=` 通常为 1-2，弱招被过滤
- [ ] 宽松局面（T>2.5）时 `candidates=` 与 v1.1 相近（过滤宽松）
- [ ] 验证：当最强招 finalScore=100，次强=72 时，T=1.0 → thresh=0.75，floor=75 → 72 < 75 被过滤，candidates 减少 1

**负向回归**：
- [ ] candidates ≠ 0（始终至少 1 个候选，最强招必然通过 thresh）
- [ ] 单招可用时行为与 v1.1 相同（candidates=1，不改变结果）

## B. v1.1 继承项重点复查

以下 v1.1 验证项在 v1.1.1 参数变化后需重点回归：

### B1. T 调整叠加正确性

- [ ] 残局（alive=2）+ 对手强化 ≥ 2：T = 2.0 × 0.60 × 0.30 ≈ 0.36 → 钳制到 0.40
- [ ] 残局（alive=2）+ switchP > 65：T = 2.0 × 1.25 × 0.30 = 0.75（钳制前 0.75，不触发下限）
- [ ] T 始终在 [0.4, 4.0] 范围内

### B2. 权重分布（用更低 T 对比 v1.1）

- [ ] T=0.5（残局典型值），movepow 比 3:1 时：强招权重 100，弱招约 ≤20
- [ ] T=0.5 时 `candidates=1` 的频率应明显高于 v1.1（过滤更激进）
